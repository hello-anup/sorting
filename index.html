🫧 Bubble Sort

The simplest sorting algorithm. It works by repeatedly stepping through the list, comparing adjacent elements, and swapping them if they are in the wrong order. The biggest unsorted elements "bubble up" to the end — just like air bubbles rising in water!

📖 How Bubble Sort Works — Step by Step

Real-life analogy: Imagine sorting a row of students by height. You walk down the row, and whenever you see a taller student before a shorter one, you ask them to swap. After one full walk (pass), the tallest student will be at the end. Repeat until everyone is sorted!

Pass 1 Example — Array: [5, 3, 8, 1, 9]
Compare 5,3 → swap → [3,5,8,1,9]
Compare 5,8 → no swap → [3,5,8,1,9]
Compare 8,1 → swap → [3,5,1,8,9]
Compare 8,9 → no swap → [3,5,1,8,9] ✓ 9 is sorted!

Key idea: After each pass, the largest unsorted element "bubbles up" to its final position. So after Pass 1, the largest element is sorted. After Pass 2, the second largest is sorted. We need n-1 passes total.

Optimization: If a full pass happens without any swaps, the array is already sorted! We can stop early. This is why Bubble Sort has O(n) best case for already-sorted arrays.

⏱ Time Complexity

🟢 Best CaseΩ(n)
🟡 Average CaseΘ(n²)
🔴 Worst CaseO(n²)
Why O(n²)? We have n passes, and in each pass we do up to n-1 comparisons. So total = n × (n-1) ≈ n². It's like a double loop!

Why O(n) best? If array is already sorted, we do one pass with zero swaps and stop. Just n-1 comparisons!

💾 Space Complexity

Auxiliary SpaceO(1)
In-place algorithm: We only need a temporary variable for swapping. No extra arrays needed!

What is O(1) space? It means the amount of extra memory used doesn't grow with the input size. Always constant!

Stable: Yes! Equal elements keep their original order because we only swap when strictly greater.

🎯 Selection Sort

Selection Sort divides the array into a sorted and an unsorted part. It repeatedly selects the minimum element from the unsorted part and moves it to the sorted part. Like organizing a deck of cards by always picking the lowest card!

📖 How Selection Sort Works

Real-life analogy: You have a shuffled hand of numbered cards. Look through ALL cards, find the smallest number, and put it first. Then look through the remaining cards, find the smallest, put it second. Repeat until done!

Example — Array: [64, 25, 12, 22]
Pass 1: Find min in [64,25,12,22] → min=12 → swap 64↔12 → [12,25,64,22]
Pass 2: Find min in [25,64,22] → min=22 → swap 25↔22 → [12,22,64,25]
Pass 3: Find min in [64,25] → min=25 → swap 64↔25 → [12,22,25,64]
Done! → [12,22,25,64] ✓

Key difference from Bubble Sort: Selection Sort makes only ONE swap per pass (the minimum to its position), while Bubble Sort may make many swaps per pass. So Selection Sort has fewer swaps but same comparisons.

⏱ Time Complexity

🟢 Best CaseΩ(n²)
🟡 Average CaseΘ(n²)
🔴 Worst CaseO(n²)
Always O(n²)! Even if sorted, we still scan the entire unsorted portion to FIND the minimum. We can't skip. That's why all three cases are n².

💾 Space & Stability

SpaceO(1)
StableNo
Why not stable? Swapping the minimum to the front can change the relative order of equal elements.

Good news: Only n-1 swaps total! Great for systems where swaps are expensive (like writing to slow storage).

🃏 Insertion Sort

Builds the sorted array one element at a time. Takes each new element and inserts it into the correct position among already-sorted elements. Exactly how you sort playing cards in your hand!

📖 How Insertion Sort Works

Card analogy: Pick up cards one by one. When you pick up a new card, slide it left into its correct position among your already-sorted cards. You shift cards right to make space!

Example — Array: [5, 2, 4, 6, 1]
i=1, key=2: 5>2, shift 5 right → insert 2 → [2,5,4,6,1]
i=2, key=4: 5>4, shift 5 right → 2<4, stop → insert 4 → [2,4,5,6,1]
i=3, key=6: 5<6, no shift → insert in place → [2,4,5,6,1]
i=4, key=1: shift 6,5,4,2 right → insert 1 first → [1,2,4,5,6] ✓

Why it's great for nearly sorted arrays: If an element is already close to its correct position, we only shift a few times. In the best case (already sorted), we never shift at all — just n-1 comparisons!

⏱ Time Complexity

🟢 Best CaseΩ(n)
🟡 Average CaseΘ(n²)
🔴 Worst CaseO(n²)
Worst case (reverse sorted): Every element must shift all the way left. Element at position k shifts k times. Total shifts: 0+1+2+...+(n-1) = n(n-1)/2 = O(n²).

Best case (already sorted): Every new element is already larger than the sorted portion. No shifts, just n-1 comparisons = O(n)!

💾 Space & Stability

SpaceO(1)
StableYes
Online algorithm: Can sort a list as it receives it, one element at a time! Great for streaming data.

Adaptive: Runs faster on partially sorted arrays. Used in hybrid algorithms like Timsort (Python's built-in sort)!

⚡ Quick Sort

A powerful divide-and-conquer algorithm. It picks a "pivot" element and partitions the array into elements smaller than pivot (left) and larger (right). Then recursively sorts each side. Very fast in practice!

📖 How Quick Sort Works

Analogy: Imagine sorting books by choosing one book as the reference. All books cheaper go left, more expensive go right. Then do the same for each group independently!

Example — Array: [3, 6, 8, 10, 1, 2, 1]
Choose pivot = 1 (last element)
Partition: elements ≤ 1 go left → [1,1] | [pivot placed] | [3,6,8,10,2]
Recursively sort left → [1,1] (already done!)
Recursively sort right → choose pivot=2... and so on
Final → [1,1,2,3,6,8,10] ✓
Why is it fast? A good pivot splits the array in half each time. We go from n elements → n/2 → n/4 → ... That's log(n) levels. Each level does O(n) work total. So n × log(n) total!

⏱ Time Complexity

🟢 Best CaseΩ(n log n)
🟡 Average CaseΘ(n log n)
🔴 Worst CaseO(n²)
Worst case happens when: Pivot is always the smallest or largest element (already sorted array with bad pivot choice). Recurrence: T(n) = T(n-1) + O(n) = O(n²).

Fix: Use random pivot selection or median-of-three to avoid worst case in practice.

💾 Space & Stability

Space (avg)O(log n)
Space (worst)O(n)
StableNo
Recursion stack: Each recursive call uses stack space. With good pivot, the recursion depth is log(n). With bad pivot (worst case), depth is n.

In practice: Quick Sort is often the fastest! Most standard library sorts use it internally.

🔀 Merge Sort

A classic divide-and-conquer algorithm that divides the array in half, recursively sorts each half, then merges the two sorted halves. Guaranteed O(n log n) in ALL cases. The gold standard for stable sorting!

📖 How Merge Sort Works

Analogy: Two sorted piles of cards. To merge them: look at the top card of each pile, pick the smaller one, place it in your result pile. Repeat until both piles are empty. Now imagine first splitting one big pile into many single-card piles, then merging them up!

Array: [38, 27, 43, 3]
Split: [38,27] | [43,3]
Split: [38] [27] | [43] [3]
Merge [38]+[27] → compare 38>27 → [27,38]
Merge [43]+[3] → compare 43>3 → [3,43]
Merge [27,38]+[3,43]: 3<27→3, 27<43→27, 38<43→38, 43→43
Result: [3,27,38,43] ✓
Recurrence relation: T(n) = 2T(n/2) + O(n). We solve two halves, each of size n/2, then spend O(n) time merging. By Master Theorem: T(n) = O(n log n). Always!

⏱ Time Complexity

🟢 Best CaseΩ(n log n)
🟡 Average CaseΘ(n log n)
🔴 Worst CaseO(n log n)
Always O(n log n)! The splitting always creates log(n) levels. Each level does exactly n comparisons total (across all merges on that level). So log(n) levels × n work = n log n. No bad case!

💾 Space & Stability

SpaceO(n)
StableYes
Extra space needed! Merging requires temporary arrays. Total extra space = O(n). This is the tradeoff vs Quick Sort.

Stable: Equal elements maintain their original order. Essential for multi-key sorting!

Best for: Linked lists (no random access needed), external sorting, any time stability matters.
